# Durak Terminal Game PRD

## 1. Overview

- **Problem summary:** build a terminal-first Durak game that starts as a simple offline CLI against a bot, but can grow into a comfortable TUI, multiplayer server, configurable rule engine, and AI experimentation platform.
- **Proposed solution:** implement the game around a clean Go domain core for cards, rules, turns, validation, bot decisions, and match events. The first release exposes this through a minimal Linux CLI; later releases can reuse the same core through a Bubble Tea TUI, Wish daemon/server mode, SSH sessions, and bot/AI strategy adapters.
- **Why now:** starting from a small CLI allows the core rules and abstractions to be proven before investing in terminal UI polish or network play.

## 2. Target Users

- **Primary user:** a Linux terminal user who wants to play Durak locally against a bot.
- **Secondary users:** future remote players connecting through SSH; developers experimenting with bot strategies and AI agents; operators hosting a private terminal game server.
- **Core user pain points:** most terminal card games are either too primitive to extend or too overbuilt before the rule model is stable. The product should keep early UX simple while avoiding architectural dead ends.

## 3. Goals and Success Metrics

- **Primary goal:** ship a playable, testable CLI Durak MVP with a domain model that supports later TUI, multiplayer, configurable rules, stats, and bot strategy evolution.
- **Success metrics:**
  - A full offline game can be completed against a bot without manual state correction.
  - Invalid moves are rejected with clear reasons.
  - Core game logic is covered by automated tests for shuffle/deal, attack, defense, taking cards, round completion, replenishment, and win/loss conditions.
  - CLI and bot logic depend on the game core through explicit interfaces rather than direct state mutation.
- **Product direction:**
  - Early implementation should stay deliberately small.
  - Target architecture should still make room for configurable rules, multiple seats, persistent match history, SSH play, ratings, currency, TUI, and AI-driven bot strategies.
- **Non-goals:**
  - No polished TUI in the first implementation milestone.
  - No real-money mechanics.
  - No persistence of interrupted games in v1.
  - No public matchmaking or account system in v1.

## 4. MVP Scope

### In Scope

- Offline CLI game: one human player versus one bot.
- Classic 36-card Durak deck.
- Configurable rule profile with sensible defaults for classic play.
- Domain-level validation of legal moves.
- Simple deterministic bot strategy layer.
- Match event log suitable for future history/statistics.
- Internal separation between game core, player input, bot decision-making, and presentation.

### Out of Scope

- Full-screen TUI.
- SSH multiplayer daemon.
- Rating, currency balances, long-term statistics UI.
- AI/LLM-driven bot decisions.
- Save/resume of interrupted games.
- DSL implementation for strategies.

## 5. Functional Requirements

### Feature: Configurable Rule Profile

- **Description:** the game must be created from a rule profile rather than hard-coded assumptions.
- **User value:** supports classic Durak first while leaving room for table-specific variants.
- **Acceptance criteria:**
  - Target rules can define deck size/layout, initial hand size, redeal conditions, trump-card selection constraints, first-attacker policy, turn direction, whether throwing in is allowed, who may throw in, throw-in limits, whether transfer is enabled, whether the first attack can be transferred, table seat limits, and future move time limits.
  - MVP uses a built-in classic preset in code rather than external configuration.
  - Unsupported future options can be represented in config but fail clearly if selected before implementation.

### Feature: Default Rule Preset

- **Description:** the product has a default rule preset that reflects the desired house rules while keeping each rule configurable in the target system.
- **User value:** gives the first implementation concrete behavior and prevents future rule configuration from being designed too narrowly.
- **Acceptance criteria:**
  - The default deck has 36 cards and each player receives 6 cards.
  - If any player receives 5 or more cards of the same suit in the initial 6-card hand, the deck is reshuffled and the game is redealt.
  - The trump indicator card cannot be an ace. If an ace is selected as the indicator, player hands remain unchanged; only the undealt stock is reshuffled and a new trump indicator is selected from that stock.
  - The first attacker is the player holding the lowest trump card among all player hands.
  - If no player has a trump card after the deal, the first attacker is selected randomly.
  - In consecutive matches within a series, the first attacker is the player seated immediately before the previous match loser.
  - With more than 2 players, turn order proceeds clockwise.
  - Throw-ins are allowed from all players, not only neighboring seats.
  - The player who started the attack gets the first throw-in opportunity.
  - A throw-in is legal only when its rank already appears on the table.
  - The first successful defense is limited to 5 attacking cards total.
  - If a defender takes cards, that round does not count as the first successful defense.
  - After the first successful defense, there is no fixed throw-in count cap beyond rule legality and available cards.
  - If the defender partially defends and then must take, other players may throw in additional legal cards before the defender picks up.
  - The first attack of the match cannot be transferred.
  - Later transfers are allowed by adding a card of the same rank according to standard transfer rules.
  - Cheating mechanics are deferred to v2.
  - Move time limits are part of the target rule model but are not implemented in the first CLI version.

### Feature: Core Game Engine

- **Description:** the engine owns match state, turn state, legal move validation, state transitions, and outcome detection.
- **User value:** all UI, bot, multiplayer, and AI surfaces can use the same rules.
- **Acceptance criteria:**
  - The engine exposes legal actions for the active player.
  - The engine rejects invalid card selections and invalid phase transitions.
  - Attack, defense, take, pass/done, replenishment, next attacker selection, and game-end conditions are deterministic and testable.

### Feature: CLI Offline Play

- **Description:** a command-based interface allows a human to play against a bot in the terminal.
- **User value:** earliest playable version with low UI complexity.
- **Acceptance criteria:**
  - The user can start a new game with default rules.
  - The CLI displays player hand, visible table cards, trump card/suit, deck count, and available commands.
  - Commands provide actionable error messages when invalid.
  - A complete game can be played without restarting the process.

### Feature: Bot Strategy Adapter

- **Description:** bot decisions are requested through an interface independent of the game engine.
- **User value:** simple algorithmic bot now; future difficulty levels, DSL strategies, and AI-backed decisions later.
- **Acceptance criteria:**
  - MVP bot can choose attack, defense, take, and pass/done actions.
  - Bot receives a bounded decision context, not mutable engine internals.
  - Bot output is validated by the same engine path as human input.
  - Strategy behavior can be swapped without editing core rule code.

### Feature: Raw AI Command Tester

- **Description:** a headless AI-facing controller can receive a seat-scoped prompt and return textual player commands through the same command language used by CLI players.
- **User value:** AI agents can stress parser, validation, and match stability with non-linear, occasionally invalid input before provider-specific AI integrations exist.
- **Acceptance criteria:**
  - Raw AI commands are parsed through a shared text-command adapter, not by duplicating CLI parsing logic.
  - Invalid AI commands are captured as controller errors with enough context for debugging.
  - External raw-command wrappers can be connected through a documented stdin/stdout protocol without introducing provider SDKs into core gameplay.
  - Raw AI testing does not allow AI adapters to mutate game state directly.
  - The design leaves room for future structured AI players that return action identifiers or decisions without using the raw command parser.

### Feature: Match Events and History Foundation

- **Description:** important state changes are emitted as structured match events.
- **User value:** future replay, statistics, rating, currency, and AI training can use the same source of truth.
- **Acceptance criteria:**
  - Events include game start, deal, attack, defense, take, pass/done, refill, round end, and match end.
  - Events include enough data to reconstruct public match progression.
  - First core implementation may keep events in memory only.
  - Durable match persistence is required for the target product, but should be designed after the core game loop exists.

## 6. User Experience Notes

- **Key flows:**
  - Start game.
  - Read visible game state.
  - Select cards by stable hand indexes or short card codes.
  - Confirm attack/defense/take/done actions.
  - See bot move and continue.
- **Important UX constraints:**
  - First CLI should prefer clarity over visual polish.
  - Card labels must be compact and unambiguous.
  - Error messages should explain what move is legal now.
  - Later TUI should remain practical rather than animation-heavy.

## 7. Data and Domain Model

- **Core entities:**
  - Card, suit, rank, deck, trump.
  - Player, hand, seat.
  - Match, round, turn phase, table pair.
  - Rule profile.
  - Action, legal action, validation result.
  - Bot strategy, decision context.
  - Match event.
  - Future: account, rating, currency ledger, game history, replay, AI training dataset.
- **Important fields:**
  - Match id, rule profile id/version, player ids, seat order, active attacker/defender, deck state, discard pile, hands, table cards, event sequence number.
- **Relationships or lifecycle states:**
  - Match moves from setup to active rounds to completed.
  - Round moves through attack, defense, throw-in/continue, take or beat, refill, next round.
  - Target multiplayer model uses `seats[]`; 36-card tables should be limited to at most 6 seats.
  - First playable implementation may hard-code one human seat and one bot seat while keeping the future seat model in mind.
  - Rule profile should be expressive enough to represent the default preset's redeal, trump-selection, throw-in, transfer, and timing policies.

## 8. Technical Considerations

- **Platform targets:** Linux terminal first.
- **Expected integrations:** none for MVP. Future integrations may include SSH daemon transport, persistent storage, AI decision service, and exportable replay/training data.
- **Known constraints:**
  - Keep MVP small, but do not entangle rules with CLI rendering.
  - Avoid irreversible assumptions about player count, transport, or bot implementation.
  - Do not introduce real-money flows.
- **Configuration direction:**
  - First implementation uses built-in presets.
  - Target product supports per-match configuration.
- **Stack direction:**
  - Go is the primary implementation language.
  - Charm Bubble Tea is the target TUI framework.
  - Charm Wish is the target SSH-hosted terminal application layer.
  - Rust/Ratatui and Python/Textual are not primary stack choices, but the domain model should remain isolated enough that other future clients can be built if needed.

## 9. Security and Privacy

- **Authentication needs:** none for local CLI MVP.
- **Authorization or roles:** none for local CLI MVP. Future daemon mode will need user/session identity and possibly admin/operator controls.
- **Sensitive data handling:** no sensitive data in MVP. Future ratings, currency balances, and player history should be treated as user data.
- **Major risks:**
  - SSH daemon abuse if exposed publicly.
  - Cheating or state tampering in multiplayer mode.
  - Bot/AI prompt or tool abuse if AI decisions are added later.

## 10. Delivery Plan

- **Suggested milestones:**
  - M0: PRD, stack choice, architecture sketch.
  - M1: game core, rule profile, tests.
  - M2: CLI offline player versus simple bot.
  - M3: event log and local match history foundation.
  - M4: TUI prototype using the same core.
  - M5: daemon/SSH multiplayer prototype.
  - M6: ratings, internal currency, strategy DSL, and AI bot experiments.
- **Dependencies:**
  - Agreement on remaining default-rule edge cases.
  - Agreement on how much configurability is represented in MVP versus deferred as schema only.
- **Operational or launch constraints:** local Linux terminal first; server hosting only after the game loop is stable.

## 11. Risks and Mitigations

- **Risk:** overbuilding for future multiplayer and AI before a fun local game exists.
  - **Why it matters:** complexity can slow down the playable MVP.
  - **Mitigation:** expose extension points, but implement only local CLI and simple bot first.

- **Risk:** rule flexibility makes the core hard to reason about.
  - **Why it matters:** Durak variants affect legal moves and round transitions.
  - **Mitigation:** define a default rule profile and add variant behavior behind explicit capability checks.

- **Risk:** future stats and AI training need better event data than the MVP records.
  - **Why it matters:** bad history format can limit later analysis.
  - **Mitigation:** emit structured match events from the beginning, even if persistence is simple.

## 14. Assumptions

- MVP means local Linux CLI, one human against one bot.
- The first implementation should favor a built-in classic preset over external config.
- Go is the primary stack, with Bubble Tea and Wish as the target terminal UI and SSH layers.
- Target rules are described through per-match configuration.
- Target multiplayer state uses seats with a maximum of 6 seats for 36-card games.
- Persistent match history is mandatory for the target product, but not needed before the core game is implemented.
- Rating and currency are separate mechanics.
- The default rule preset should be represented explicitly even if the first playable CLI implements only the subset needed for a small 2-player offline loop.
- TUI, SSH multiplayer, ratings, currency, DSL, and AI are future milestones, not MVP blockers.
- No real-money mechanics will be introduced.
